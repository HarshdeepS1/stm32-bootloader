// project roadmap

STM32 Bootloader
+ HIL Test Framework

A phase-by-phase plan for building a bare-metal OTA bootloader and hardware-in-the-loop test harness from scratch.

6 Phases
6–8 wks Total time
2–3 hrs Per day
2 Projects
Phase 0 Toolchain & "Hello World" Foundation
3–5 days

Before writing a bootloader, prove your tools actually work end-to-end.

  • 🔧Install and configure STM32CubeIDE; verify ST-Link is recognized by Windows
  • 💡Blink an LED on the Nucleo board — the embedded "hello world"
  • 📡Send "Hello" over UART and read it in PuTTY on your PC
  • 📖Read the STM32 reference manual sections on memory layout and flash
✅  LED blinks + UART prints to your terminal
Phase 1 Memory Layout & Linker Scripts Project 1
4–6 days

The hardest concept to grasp — but everything else depends on it.

  • 🗺️Learn how flash memory is divided into regions (bootloader vs app)
  • 📄Read and modify a linker script (.ld file) to place code at specific addresses
  • 🔢Understand the vector table and why the app must have its own copy
  • 🧪Write a minimal app that lives at a custom address and jump to it manually
✅  Bootloader code that jumps to a hardcoded app address
Phase 2 UART Receive Protocol Project 1
5–7 days

Teach the bootloader to receive a firmware file over the wire.

  • 📦Implement XMODEM or a simple custom frame protocol over UART
  • 🐍Write a Python script on your PC to send a binary file over serial
  • 🔄Receive packets, acknowledge them, handle errors and retransmission
  • 💾Store received bytes in RAM (not flash yet) and verify the count is correct
✅  PC sends a file, STM32 receives all bytes correctly
Phase 3 Flash Write + Verification Project 1
5–7 days

The scariest part — writing to memory that could brick your chip if done wrong.

  • 🔓Learn the flash erase → unlock → write sequence (mandatory order)
  • ✍️Erase the application sector and write received bytes to flash
  • 🔐Add CRC32 checksum verification so corrupt firmware is rejected
  • ↩️Implement a fallback: if verification fails, keep the old firmware
✅  Bootloader flashes new firmware and rejects corrupt images
Phase 4 HIL Test Harness (Python) Project 2
6–8 days

Build the automated test system that proves your bootloader actually works.

  • 🐍Write a Python class to control UART and reset the board programmatically
  • 🧪Write test cases: valid firmware update, corrupt image, wrong address
  • 📊Add pass/fail reporting with human-readable output logs
  • ⚡Hook into GPIO pins to simulate power cycles and trigger board reset
✅  Run one command, get a full test report with pass/fail
Phase 5 Integration & Polish Both
4–5 days

Turn a working prototype into something you can show anyone.

  • 🔁Full end-to-end test: send firmware from PC → flash → boot → HIL validates it
  • 📝Write a clean README explaining what it is and how to use it
  • 🌿Clean up Git history, structure branches properly
  • 🎥Record a short demo — great for a portfolio or resume
✅  A complete, documented, demo-ready project on GitHub

// things to keep in mind

Phase 1 will feel the hardest

Memory layout requires understanding how a computer actually stores and runs code — things hidden from you in normal software dev. Don't rush it.

You will brick the board

In Phase 3, writing bad data to flash will happen. It's fine — the Nucleo can almost always be recovered. We'll cover recovery procedures when we get there.

Projects overlap

You start writing Python (Project 2) during Phase 2 to send files over serial. By Phase 4 it becomes a formal test harness. They grow together.

Getting stuck = learning

A day spent debugging a linker script is a good day. Don't measure progress by lines of code written — measure it by concepts understood.
